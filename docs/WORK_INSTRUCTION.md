## 1. วัตถุประสงค์

เอกสารฉบับนี้กำหนดขั้นตอนมาตรฐานในการใช้งานโปรแกรม **appQC (The Precision Ledger)** เพื่อ:

- บันทึกผลการสุ่มตรวจคุณภาพสินค้าขาเข้า
- แจ้งเตือนผ่านอีเมลโดยอัตโนมัติเมื่อพบสินค้าสถานะ Reject
- เก็บประวัติการแก้ไขเพื่อการตรวจสอบย้อนกลับ

## 2. ขอบเขต

ใช้สำหรับพนักงาน QC, หัวหน้า QC และ Admin ระบบของบริษัท Comets Intertrade ทุกคนที่เข้าใช้งานโปรแกรม appQC ที่เว็บไซต์ **https://web-mocha-three-44.vercel.app**

## 3. คำจำกัดความ

| คำศัพท์ | คำอธิบาย |
|---|---|
| QC Order / ใบบันทึก QC | รายการบันทึกผลตรวจ 1 รายการ (เช่น `QC26050001`) |
| Lot / ล็อต | ชุดสินค้าที่รับเข้า ระบุด้วย Lot No. |
| NCR | รายงานสินค้าไม่ผ่าน — สร้างอัตโนมัติเมื่อสถานะเป็น Reject |
| Accept / ยอมรับ | ผ่านการตรวจ ไม่มีของเสีย |
| Accept Lot / ยอมรับล็อต | ผ่านการตรวจโดยมีของเสียบางส่วนแต่ยังอยู่ในเกณฑ์ |
| Reject / ปฏิเสธ | ไม่ผ่านการตรวจ ส่งคืนผู้จำหน่าย |
| % ของเสีย | (จำนวนของเสีย ÷ จำนวนตรวจสอบ) × 100 |

### บทบาท (Roles)

| Role | ชื่อเรียก | สิทธิ์โดยสรุป |
|---|---|---|
| `operator` | พนักงานหน้างาน | บันทึก QC, ดูและแก้ไข Order ของตนเอง |
| `qc_admin` | Admin ตรวจงาน | สิทธิ์ของ Operator + อนุมัติ + จัดการมาสเตอร์ Supplier/Defect/Brand + ส่ง Test mail |
| `admin` | Admin ระบบ | สิทธิ์ของ QC Admin + จัดการผู้ใช้ + จัดการผู้รับ Email |
| `viewer` | ผู้ดูข้อมูล | ดูประวัติได้อย่างเดียว |

## 6. Operator — บันทึกผล QC

### 6.1 ภาพรวมขั้นตอน

```
เข้าสู่ระบบ → บันทึก QC → กรอกข้อมูลสินค้า → เลือก Defect (ถ้ามี) → กดบันทึก
                                                                       ↓
                                                  Popup สรุปผล → ดาวน์โหลด NCR (กรณี Reject)
```

### 6.2 ขั้นตอนละเอียด

#### ขั้นตอนที่ 1: เปิดหน้าบันทึก QC

คลิกแท็บ **"บันทึก QC"** ที่แถบเมนูด้านบน

![ภาพ 6.2.1 — แถบเมนูด้านบนเห็นแท็บ บันทึก QC](docs.screenshots.AppQC/6.2.1.png)

*📸 ภาพ 6.2.1 — แถบเมนูด้านบนเห็นแท็บ "บันทึก QC"*

#### ขั้นตอนที่ 2: กรอกข้อมูลส่วนหัว

| ฟิลด์ | จำเป็น | คำอธิบาย |
|---|---|---|
| **Project Brief No.** | ไม่บังคับ | กรอกหากมีโครงการที่อ้างอิง |
| **วันที่รับสินค้า** | ✅ | วันที่รับสินค้าเข้ามา |
| **วันที่ตรวจ** | ✅ | วันที่สุ่มตรวจคุณภาพ (ค่าเริ่มต้น = วันนี้) |
| **เลข Order (Preview)** | อัตโนมัติ | ขึ้นอัตโนมัติเมื่อเลือกวันที่ตรวจ (เช่น `QC26050015`) |

![ภาพ 6.2.2 — ส่วนหัวฟอร์ม: Project Brief, วันที่รับ, วันที่ตรวจ, เลข Order](docs.screenshots.AppQC/6.2.2.png)

*📸 ภาพ 6.2.2 — ส่วนหัวฟอร์ม: Project Brief, วันที่รับ, วันที่ตรวจ, เลข Order*

#### ขั้นตอนที่ 3: กรอกข้อมูลสินค้า

1. กรอก **SAP Code** → ระบบจะดึงรายละเอียดสินค้า / กลุ่มสินค้า / Brand / Sales / SCM มาแสดงโดยอัตโนมัติ

![ภาพ 6.2.3 — กรอก SAP Code แล้วเห็นรายละเอียดสินค้าขึ้นอัตโนมัติ](docs.screenshots.AppQC/6.2.3.png)

*📸 ภาพ 6.2.3 — กรอก SAP Code แล้วเห็นรายละเอียดสินค้าขึ้นอัตโนมัติ*

#### ขั้นตอนที่ 4: เลือก Supplier

1. คลิก dropdown **"รหัสผู้จัดจำหน่าย / Sup Code"**
2. รายการจะแบ่งกลุ่มเป็น: **━━━ Import (xx) ━━━** และ **━━━ Local (xx) ━━━**
3. แต่ละตัวเลือกแสดงในรูปแบบ `sap_code/sup_code` (เช่น `10000138/12Y`) พร้อมชื่อ Supplier ในช่องถัดไป
4. เลือก Supplier ที่ตรงกับสินค้า

![ภาพ 6.2.4 — Dropdown Sup Code เปิดแสดงกลุ่ม Import / Local](docs.screenshots.AppQC/6.2.4.png)

*📸 ภาพ 6.2.4 — Dropdown Sup Code เปิดแสดงกลุ่ม Import / Local*

#### ขั้นตอนที่ 5: กรอก Lot และจำนวน

| ฟิลด์ | จำเป็น | ตัวอย่าง |
|---|---|---|
| **Lot No.** | ✅ | `INV23-CWZ106` |
| **จำนวนรับ** | ✅ | `75600` |
| **จำนวนตรวจสอบ** | ✅ | `500` |

![ภาพ 6.2.5 — กรอก Lot No. + จำนวนรับ + จำนวนตรวจสอบ](docs.screenshots.AppQC/6.2.5.png)

*📸 ภาพ 6.2.5 — กรอก Lot No. + จำนวนรับ + จำนวนตรวจสอบ*

#### ขั้นตอนที่ 6: เพิ่มรายการของเสีย (เฉพาะ Accept Lot หรือ Reject)

1. พิมพ์ค้นหาด้วย **รหัส** หรือ **ชื่ออาการ** ในช่องค้นหา
2. ✅ ติ๊กเลือก **หลายอาการ** ในกลุ่ม Severity เดียวกันได้
3. คลิก **"เพิ่มในรายการ"** — อาการที่เลือกจะรวมเป็น **1 แถว**
4. เลือก **ระดับความรุนแรง** — Critical / Major / Minor
5. กรอก **จำนวน** ของเสียที่พบ
6. แนบ **รูปภาพ** ของเสียได้ไม่เกิน **3 รูป** ต่อแถว (คลิกไอคอนกล้อง)

![ภาพ 6.2.6 — ค้นหา Defect และติ๊กเลือกหลายรายการ](docs.screenshots.AppQC/6.2.6.png)

*📸 ภาพ 6.2.6 — ค้นหา Defect และติ๊กเลือกหลายรายการ*

![ภาพ 6.2.7 — แถวรายการ Defect แสดง Severity / จำนวน / รูปแนบ](docs.screenshots.AppQC/6.2.7.png)

*📸 ภาพ 6.2.7 — แถวรายการ Defect แสดง Severity / จำนวน / รูปแนบ*

#### ขั้นตอนที่ 7: ตรวจสอบ % ของเสียและสถานะ

- **% ของเสีย** คำนวณอัตโนมัติแสดงที่มุมขวาบน
- **สถานะ** กำหนดอัตโนมัติตามผล:
  - `0%` → **Accept** (เขียว)
  - มีของเสียแต่อยู่ในเกณฑ์ → **Accept Lot** (เหลือง)
  - เกินเกณฑ์ → **Reject** (แดง)

![ภาพ 6.2.8 — แสดง % ของเสียและ Status chip](docs.screenshots.AppQC/6.2.8.png)

*📸 ภาพ 6.2.8 — แสดง % ของเสียและ Status chip*

#### ขั้นตอนที่ 8: เอกสารต้นฉบับและหมายเหตุ

1. เลือก **เอกสารต้นฉบับ / Original Documents** — เลือกผู้ถือเอกสารต้นฉบับ: คุณอู๋ / WH / PD / SCM / Custom
2. กรอก **หมายเหตุ** (ถ้ามี)

![ภาพ 6.2.9 — Dropdown เอกสารต้นฉบับ + ช่องหมายเหตุ](docs.screenshots.AppQC/6.2.9.png)

*📸 ภาพ 6.2.9 — Dropdown เอกสารต้นฉบับ + ช่องหมายเหตุ*

#### ขั้นตอนที่ 9: บันทึก

1. คลิกปุ่ม **"บันทึก"**
2. รอ Upload รูปภาพ (มีแถบโหลด)
3. **Popup สรุปผล** ขึ้นแสดงข้อมูลทั้งหมด + เลข Order (เช่น `QC26050015`)
4. **กรณีสถานะ Reject** ระบบจะ:
   - ส่ง Email พร้อมแนบไฟล์ NCR PDF ไปยังผู้รับทั้งหมดที่เปิดสถานะรับ Email โดยอัตโนมัติ
   - Status chip ใน Popup จะเปลี่ยนเป็น: `กำลังเตรียม → กำลังส่ง → ✅ ส่งสำเร็จ` (หรือ ❌ ส่งไม่สำเร็จ)
   - คลิก **"ดาวน์โหลด NCR PDF"** เพื่อบันทึกไฟล์ลงเครื่อง
5. คลิก **"ปิด"** → ฟอร์ม reset พร้อมรับ Order ถัดไป

![ภาพ 6.2.10 — Success Popup หลังบันทึก Reject](docs.screenshots.AppQC/6.2.10.png)

*📸 ภาพ 6.2.10 — Success Popup หลังบันทึก Reject (เห็นเลข Order, Status chip, ปุ่ม Download NCR)*

4. คลิก **"ดาวน์โหลด NCR"** เพื่อ Download NCR PDF ซ้ำได้ (เฉพาะ Order ที่สถานะ Reject)

4. หาก Order เคย **ผ่านการอนุมัติแล้ว** การอนุมัติจะถูกล้าง ต้องอนุมัติใหม่
5. การแก้ไขทุกครั้งจะถูกบันทึกใน Log (`qc_order_edit_log`) — เก็บวันเวลา + ผู้แก้ไข + รายละเอียดการเปลี่ยน

## 7. QC Admin — อนุมัติและจัดการ

### 7.1 อนุมัติรายการ Reject

1. เปิดหน้า **ประวัติ** → กรองสถานะ = `Reject`
2. คลิกที่ Order → เปิด Popup รายละเอียด
3. คลิกปุ่ม **"✓ ยืนยันการปฏิเสธ"** ในส่วน **การอนุมัติ / Approval Record**
4. ระบบบันทึกวันเวลาและชื่อผู้อนุมัติอัตโนมัติ

## 9. กระบวนการแจ้งเตือนอีเมล Reject

### 9.1 การส่งอัตโนมัติ

```
Operator บันทึก Order Reject
        ↓
Frontend สร้าง NCR PDF แบบ offscreen (html2pdf.js)
        ↓
POST /api/notify-reject พร้อมข้อมูล { order_id, pdf_base64, pdf_filename }
        ↓
Backend ตรวจสอบสิทธิ์ → สร้าง HTML email → แนบไฟล์ PDF
        ↓
Nodemailer → Gmail SMTP → ส่งไปยังผู้รับใน notification_recipients (enabled=true)
        ↓
บันทึกแถวลงตาราง notification_send_log (success / failed / skipped)
        ↓
Frontend อัปเดต Status chip ใน Success Modal
```

## 10. บันทึกและการเก็บรักษา

| ประเภทบันทึก | ที่จัดเก็บ | ระยะเวลาเก็บ |
|---|---|---|
| ข้อมูล Order | Supabase ตาราง `qc_orders` + `qc_order_details` | ไม่จำกัด |
| รูปภาพของเสีย | Supabase Storage bucket `defect-photos` | ไม่จำกัด |
| ประวัติการแก้ไข | Supabase ตาราง `qc_order_edit_log` | ไม่จำกัด |
| ประวัติการส่ง Email | Supabase ตาราง `notification_send_log` | ไม่จำกัด (UI แสดง 50 รายการล่าสุด) |
| ไฟล์ NCR PDF | สร้างขณะใช้งานฝั่ง Client | ไม่เก็บไฟล์ — ดาวน์โหลดใหม่ได้ตลอด |

## 11. การแก้ไขปัญหา

| อาการ | สาเหตุที่เป็นไปได้ | วิธีแก้ไข |
|---|---|---|
| Login ไม่ได้ ขึ้น "Invalid credentials" | รหัสผ่านผิด หรือบัญชีถูกปิด | ติดต่อ Admin ระบบให้ Reset Password |
| Error "Could not find the 'X' column" | ยังไม่ได้รัน Patch SQL ใน Database | แจ้ง Admin ระบบให้รัน Patch ที่ขาด |
| NCR PDF ตัดขอบขวา | Browser cache เก่า | กด Hard Refresh (Ctrl + Shift + R) |
| ไม่ได้รับ Email Reject | ผู้รับถูก Disable, อยู่ใน Spam, หรือ SMTP error | ตรวจ `notification_send_log` ดูสถานะ → เปิดสถานะผู้รับใหม่ หรือส่งซ้ำ |
| ไม่เห็นปุ่ม Edit | ไม่ใช่เจ้าของ Order และไม่ใช่ Admin | แจ้งผู้สร้างหรือ QC Admin ให้แก้ไขแทน |
| Dropdown Sup Code ว่างเปล่า | ยังไม่มีข้อมูล Supplier ในระบบ | ให้ Admin ระบบเพิ่มข้อมูลที่ Admin → Suppliers |
| ส่ง Test ได้ Email แต่ไม่มี PDF แนบ | Bundle เก่ายังไม่อัปเดต | Hard Refresh เบราว์เซอร์ (Ctrl + Shift + R) |

> 📞 **ช่องทางแจ้งปัญหา**: หากปัญหาไม่อยู่ในตารางข้างต้น ติดต่อ Admin ระบบที่อีเมล `data.comets@gmail.com`
